## 1) What does `cem_core.py` actually compute? Inputs and outputs.

`cem_core.py` is **domain-agnostic infrastructure + reusable physics utilities**, not ARIA-specific. It provides:

### Core dataclasses (inputs)
- **`Material`** (`@dataclass`): generic material properties and allowables.
  - Key fields (examples): `density_kg_m3`, `yield_strength_MPa`, `ultimate_strength_MPa`, `youngs_modulus_GPa`, `poisson_ratio`, etc.
  - **Key method:** `Material.allowable_stress_MPa(safety_factor: float = 2.0) -> float` (allowable stress = yield/SF).
- **`Fluid`** (`@dataclass`): fluid properties with derived `Prandtl` number.
- **`PressureLoad`** (`@dataclass`): has `pressure_Pa` and `safety_factor`.
  - **Key method:** `PressureLoad.required_wall_thickness(radius_m: float, material: Material) -> float` implements thin-wall hoop: \(t = P r SF / \sigma_{allow}\).
- **`ThermalLoad`** (`@dataclass`): defines heat flux limits and can compute required convection.
  - **Key method:** `ThermalLoad.required_h_conv() -> float` uses \(h = q / (T_{wall}-T_{fluid})\).
- **`MechanicalLoad`** (`@dataclass`): basic forces/moments.
- **`Channel`** (`@dataclass`): flow channel physics utilities (velocity, Reynolds, Dittus-Boelter, Darcy-Weisbach).

### Abstract module interface (inputs/outputs)
`cem_core.py` defines the abstract base class:
- **`class CEMModule(ABC)`**
  - `compute(self) -> Any`: “derive geometry from input parameters” and returns a geometry object.
  - `validate(self) -> bool`: returns True when module warnings list is empty.
  - `export(self, out_dir: str) -> List[str]`: returns created output file paths.

### Platform registry (orchestration)
- **`class CEMPlatform`**
  - `register(self, key: str, module: CEMModule)`
  - `run(self, key: str, out_dir: str = "output") -> Any`: calls `compute()`, then `validate()`, then `export()` and returns the module’s geometry object.
  - `run_all(self, out_dir: str = "output") -> Dict[str, Any]`
  - `summary(self)` prints counts of ok/warnings.

### Shared physics utilities (pure functions)
Notable functions used by some domains:
- `hoop_stress_Pa(pressure_Pa: float, radius_m: float, thickness_m: float) -> float`
- `min_wall_thickness(pressure_Pa: float, radius_m: float, material: Material, safety_factor: float = 2.0) -> float`
- `dittus_boelter(...)`, `reynolds(...)`, etc.

**What it “computes” in practice:** it computes **generic engineering quantities** (stress, thickness, pressure drop, h_conv) given the dataclass inputs above, and provides a standard lifecycle and logging scaffold (`compute/validate/export`) for domain modules like `ARIAModule` in `aria_cem.py`.

## 2) What does `dynamic_drop.py` model? What equations? What inputs does it need?

`aria_models/dynamic_drop.py` models a **1D dynamic drop test** as a **mass-spring-damper** with phases:

### Main API
- **`simulate_drop_test(...) -> tuple[pandas.DataFrame, dict]`**
  - Returns:
    - `df` time series with columns: `time_s`, `pos_m`, `vel_ms`, `accel_ms2`, `tension_N`, `absorber_force_N`, `phase`
    - `summary` dict with keys:
      - `arrest_distance_mm`
      - `peak_force_N`
      - `avg_force_N`
      - `trigger_fired` (bool)
      - `absorber_activated` (bool)
      - `ansi_peak_limit_N` (=8000)
      - `ansi_avg_limit_N` (=6000)
      - `ansi_distance_limit_mm` (=813)
      - `passed` (bool)

### Inputs (explicit)
`simulate_drop_test()` parameters (selected; exact names):
- `mass_kg` (default `DEFAULT_MASS_KG = 140.0`)
- `drop_height_m` (default `DEFAULT_DROP_HEIGHT_M = 0.040`)
- `trigger_g` (default `DEFAULT_TRIGGER_G = 0.7`)
- absorber model:
  - `absorber_k` (default `DEFAULT_ABSORBER_K = 30000.0`)
  - `absorber_c` (default `DEFAULT_ABSORBER_C = 2000.0`)
  - `absorber_fmax_n` (default `DEFAULT_ABSORBER_FMAX = 4000.0`)
- rope stiffness:
  - `rope_k` (default `DEFAULT_ROPE_K = 80000.0`)
- slack / numerics:
  - `slack_m` (default `DEFAULT_SLACK_M = 0.0`)
  - `dt_s` (default `DT = 0.0005`)
  - `t_max_s` (default `T_MAX = 2.0`)

### Equations / mechanics (as implemented)
Constants:
- `G = 9.81`

State variables:
- position `x` (m), velocity `v` (m/s), time `t` (s).

Phases:
1. **FREE_FALL**
   - `accel = G`, `tension_N = 0`
   - transition to `ROPE_TAUT` when `x >= total_drop` where `total_drop = drop_height_m + slack_m`
2. **ROPE_TAUT**
   - elongation: `elongation = x - total_drop`
   - rope tension: `tension_N = rope_k * elongation` if elongation > 0
   - acceleration: `accel = G - (tension_N / mass_kg)`
   - trigger condition: `accel <= -trigger_g * G` sets `trigger_fired=True` and switches to `ARREST`
3. **ARREST**
   - absorber stroke: `stroke = x - x_trigger`
   - absorber force: `absorber_force_N = absorber_k * stroke + absorber_c * v`
   - tension is set equal to absorber force: `tension_N = absorber_force_N`
   - acceleration: `accel = G - (tension_N / mass_kg)`
   - collects `arrest_forces` for averaging

Stop condition:
- exits when `v <= 0.0 and phase == "ARREST"`.

### False-trip check API
- **`simulate_false_trip_check(...) -> dict`**
  - Inputs: `mass_kg`, `accel_g` (default 0.3), `rope_k`, `trigger_g`, `dt_s`, `duration_s`
  - Computes constant-tension implied acceleration:
    - `tension_N = mass_kg * G * (1.0 - accel_g)`
    - `accel = G - (tension_N / mass_kg)`
  - **Pass criterion:** `passed = not trigger_fired`

## 3) What does `static_tests.py` check? What failure criteria does it use?

`aria_models/static_tests.py` is a **virtual static load test** for “Setup 1” and returns safety factors per load step.

### Main API
1. **`simulate_static_pawl(...) -> pandas.DataFrame`**
   - Inputs:
     - `load_steps: list[float]` (Newtons)
     - `ansi_static_n: float = 16000.0`
     - plus a set of optional geometry overrides:
       - `pawl_tip_width_mm`, `pawl_engagement_mm`, `pawl_thickness_mm`, `pawl_body_h_mm`, `pawl_arm_mm`
       - `housing_wall_mm`
       - `shaft_d_mm`, `shaft_span_mm`
   - Returns DataFrame with columns (exact column names):
     - `load_N`
     - `contact_stress_MPa`, `bending_stress_MPa`, `housing_stress_MPa`, `shaft_stress_MPa`
     - `sf_contact`, `sf_bending`, `sf_housing`, `sf_shaft`
     - `min_sf`
     - `passed` (True if `min_sf >= 2.0`)

2. **`ansi_proof_load_check(...) -> dict`**
   - Inputs:
     - `housing_wall_mm: float = HOUSING_WALL_MM`
     - `yield_mpa: float = YIELD_HOUSING_MPA`
     - `ansi_load_n: float = 16000.0`
   - Output dict keys:
     - `test` (string label)
     - `load_N`, `stress_MPa`, `yield_MPa`, `safety_factor`, `passed`
   - Pass criterion: `passed = sf >= 2.0`

### Stress models used (as implemented)
All stresses are simplified closed-form:

- Pawl contact stress:
  - `_pawl_contact_stress_mpa(force_n, tip_width_mm, engagement_mm)`:
    - `area_mm2 = tip_width_mm * engagement_mm`
    - `stress_pa = (force_n / N_PAWLS) / area_m2`

- Pawl bending stress (cantilever):
  - `_pawl_bending_stress_mpa(force_n, thickness_mm, body_h_mm, arm_mm)`:
    - `M = (force_n / N_PAWLS) * arm`
    - `I = b*h^3 / 12`
    - `sigma = M*c/I`

- Housing bearing stress at boss:
  - `_housing_wall_stress_mpa(force_n, wall_mm, boss_d_mm=20.0)`:
    - `area_mm2 = wall_mm * boss_d_mm`
    - `stress = force/area`

- Shaft bending (simply supported center load):
  - `_shaft_bending_stress_mpa(force_n, shaft_d_mm, span_mm)`:
    - `M = F*L/4`
    - `I = pi*d^4/64`
    - `sigma = M*c/I`

### Failure criteria (static_tests)
For each load step `F`:
- It computes safety factors:
  - `sf_contact = YIELD_PAWL_MPA / contact_stress_MPa`
  - `sf_bending = YIELD_PAWL_MPA / bending_stress_MPa`
  - `sf_housing = YIELD_HOUSING_MPA / housing_stress_MPa`
  - `sf_shaft = YIELD_SHAFT_MPA / shaft_stress_MPa`
- Then `min_sf = min(sf_contact, sf_bending, sf_housing, sf_shaft)`
- **Pass** for that load step is `passed = min_sf >= 2.0`

For the proof-load helper:
- `ansi_proof_load_check()` computes `stress_MPa` via `_housing_wall_stress_mpa(ansi_load_n)`
- `safety_factor = yield_mpa / stress_MPa`
- **Pass** if `safety_factor >= 2.0`

## 4) What part-level dimensions do these models need as inputs?

### A) `dynamic_drop.py` (dynamic drop test)
This model is **not directly parameterized by CAD dimensions**. It needs:
- `mass_kg` (test mass)
- `drop_height_m` / `slack_m`
- `rope_k` (rope stiffness) and absorber parameters (`absorber_k`, `absorber_c`, `absorber_fmax_n`)
- `trigger_g` threshold

If you wanted to link it to generated geometry, you’d need a mapping from geometry/material to:
- effective spring constants (`rope_k`, `absorber_k`) and damping (`absorber_c`)
- a “trigger threshold” (`trigger_g`) derived from flyweight/clutch geometry (not currently connected)

### B) `static_tests.py` (static load checks)
This model **does take explicit part dimensions**, via module-level defaults and optional overrides to `simulate_static_pawl()`.

Hardcoded default geometry constants (names in code):
- Pawl:
  - `PAWL_TIP_WIDTH_MM = 6.0`
  - `PAWL_THICKNESS_MM = 9.0`
  - `PAWL_ARM_MM = 45.0`
  - `PAWL_BODY_H_MM = 22.0`
  - `PAWL_ENGAGEMENT_MM = 3.0`
- Ratchet (note: only pitch radius and face width are declared; they’re not directly used in current stress functions):
  - `RATCHET_PITCH_R_MM = 100.0`
  - `RATCHET_FACE_W_MM = 20.0`
  - `N_TEETH = 24`
  - `N_PAWLS = 2`
- Housing:
  - `HOUSING_WALL_MM = 10.0`
  - `HOUSING_W_MM = 700.0`, `HOUSING_H_MM = 680.0`, `HOUSING_D_MM = 344.0` (width/height/depth are not directly used in the stress functions, but are still hardcoded constants)
- Shaft:
  - `SHAFT_D_MM = 20.0`
  - `SHAFT_SPAN_MM = 344.0`

Hardcoded yield strengths (names in code):
- `YIELD_PAWL_MPA = 1800.0`
- `YIELD_RATCHET_MPA = 1300.0` (not currently used in stress checks)
- `YIELD_HOUSING_MPA = 276.0`
- `YIELD_SHAFT_MPA = 1000.0`

So, **the minimal part-level geometric inputs** static_tests expects (if you supply overrides) are:
- `pawl_tip_width_mm`, `pawl_engagement_mm`, `pawl_thickness_mm`, `pawl_body_h_mm`, `pawl_arm_mm`
- `housing_wall_mm`
- `shaft_d_mm`, `shaft_span_mm`

### C) `aria_cem.py` (ARIA device-level CEM)
This model is parameterized by **`ARIAInputs`** (dataclass) and produces a full **`ARIAGeom`**.

Key geometry-like input fields (names in code):
- `brake_drum_diameter_mm`
- `rope_spool_diameter_mm` (**commented as “spool hub diameter”**)
- `housing_od_mm`
- `wall_mount_bolt_pattern_mm`
- `rope_diameter_mm`
- `max_rope_capacity_m`

Additionally, there are “design/physics requirement” inputs that are not CAD dimensions but drive the derived geometry:
- `min_hold_force_kN`, `max_arrest_force_kN`, `fall_detection_v_m_s`, `target_tension_N`
- `safety_factor_structural`, `safety_factor_fatigue`, `n_cycles_design`

**Important:** `aria_cem.py` uses both:
- **geometry constraints** from `ARIAInputs` (e.g., `brake_drum_diameter_mm`, `housing_od_mm`)
- **hardcoded “reference” dimensions** inside functions (e.g., drum width `w_mm = 60.0` inside `compute_brake_drum()`).

## 5) What does a “pass” look like from each model? What does “fail” look like?

### A) `dynamic_drop.py`
In `simulate_drop_test()`, summary `passed` is True only if all are satisfied:
- `trigger_fired == True`
- `arrest_distance_m * 1000 < limit_dist` where `limit_dist = 813.0` (mm)
- `peak_force_N < limit_peak` where `limit_peak = 8000.0` (N)
- `avg_force_N < limit_avg` where `limit_avg = 6000.0` (N)

**Fail** looks like:
- `trigger_fired == False` (never reached `accel <= -trigger_g * G`), or
- any of the ANSI limits exceeded.

In `simulate_false_trip_check()`, **pass** is `passed = not trigger_fired` under the applied `accel_g` (default 0.3g).

### B) `static_tests.py`
In `simulate_static_pawl()`:
- **Pass per load step:** `passed = (min_sf >= 2.0)`
- **Fail per load step:** `min_sf < 2.0`

In `ansi_proof_load_check()`:
- **Pass:** `safety_factor >= 2.0` at `ansi_load_n` (default 16,000 N)
- **Fail:** `< 2.0`

### C) `aria_cem.py` (`ARIAModule.validate`)
`ARIAModule.validate(self) -> bool` runs a set of checks via `self.physics_check(...)` and returns:
- **Pass** if `len(self.warnings) == 0`
- **Fail** if any warnings are added

Checks include (variable names as used in code):
- `g.predicted_peak_force_kN <= ANSI_Z359_14['max_arrest_force_kN']`
- `g.predicted_arrest_distance_m <= ANSI_Z359_14['arrest_distance_m']`
- `g.ratchet.safety_factor >= i.safety_factor_fatigue`
- `g.brake_drum.safety_factor >= i.safety_factor_structural`
- `g.clutch.safety_margin >= 3.0`
- `g.clutch.engagement_v_m_s > 0.5`
- `g.spool.capacity_m >= i.max_rope_capacity_m`

Note: it also calls `self.ok(...)` for a back-drive note (not a failure).

### D) `aria_models/state_machine.py`
This is not a “pass/fail” physical test. It outputs control decisions:
- `AriaStateMachine.step(self, inp: Inputs) -> Outputs`
- `Outputs` contains `motor_mode` and `state` (enum `State`).

The only “hard” failure-style behavior in this file is:
- If `inp.estop` is True, state becomes `State.ESTOP` and `motor_mode="OFF"`.

### E) `aria_fault_behavior.py`
This is also not a physical “pass/fail” model; it’s a **fault-response spec** and a fault injection helper:
- `FAULT_TABLE: list[FaultEntry]` defines response policies.
- `FaultSimulator.inject(fault_id, at_time_s=0.0) -> FaultEntry | None` and `FaultSimulator.active_severity() -> str | None`.

## 6) Hardcoded ARIA constants that should be in `context/aria_mechanical.md` but aren’t

`context/aria_mechanical.md` currently contains (examples): housing 700×680×344, wall 10, rope spool dia 600, brake drum dia 200, etc.

### A) `aria_models/static_tests.py` hardcodes many ARIA mechanical constants
These are explicitly declared in `static_tests.py` but not present (as such) in `context/aria_mechanical.md`:
- Pawl geometry: `PAWL_TIP_WIDTH_MM`, `PAWL_THICKNESS_MM`, `PAWL_ARM_MM`, `PAWL_BODY_H_MM`, `PAWL_ENGAGEMENT_MM`
- Shaft geometry: `SHAFT_D_MM`, `SHAFT_SPAN_MM`
- Ratchet geometry: `RATCHET_PITCH_R_MM`, `RATCHET_FACE_W_MM`, `N_TEETH`, `N_PAWLS`
- Material strengths: `YIELD_PAWL_MPA`, `YIELD_RATCHET_MPA`, `YIELD_SHAFT_MPA` (yield of housing 6061 matches context’s material but the numeric yield is not in context)

If you want ARIA-OS/CAD to feed these checks automatically, these should move to a single source of truth (likely `context/aria_mechanical.md`, or a new `context/aria_materials.md` for yield strengths/hardness).

### B) `aria_models/dynamic_drop.py` hardcodes test fixture + model parameters
These are not in `context/aria_mechanical.md` and are more “test rig / model” constants than mechanical geometry:
- `DEFAULT_MASS_KG = 140.0`
- `DEFAULT_DROP_HEIGHT_M = 0.040`
- `DEFAULT_TRIGGER_G = 0.7`
- `DEFAULT_ROPE_K = 80000.0`
- absorber parameters (`DEFAULT_ABSORBER_K`, `DEFAULT_ABSORBER_C`, `DEFAULT_ABSORBER_FMAX`)
- ANSI limits used here: 813mm, 8000N, 6000N

These likely belong in a separate context document (e.g. `context/aria_test_standards.md`) rather than `aria_mechanical.md`.

### C) `aria_cem.py` duplicates and/or hardcodes values that overlap with `context/aria_mechanical.md`
Examples:
- `ANSI_Z359_14` dict is hardcoded (limits and test mass).
- Materials dicts `MATERIAL_6061_T6`, `MATERIAL_4140_STEEL` are hardcoded (yield, E, density).
- In-function constants:
  - `compute_brake_drum()`: `w_mm = 60.0` (axial width reference)
  - `compute_ratchet()`: `pressure_angle_deg = 26.0`, `module_mm = 3.0`, `Y_lewis = 0.154 - 0.912/20`
  - `compute_centrifugal_clutch()`: `v_climb_normal = 0.3`, `r_fw = R_drum * 0.60`, `n_fw = 3`, `F_spring_target = 8.0`, engagement factor `0.85`
  - `compute_rope_spool()`: `n_layers = 6`, rope linear mass `m_rope = L_rope * 0.065`
  - `compute_motor()`: `motor_peak_Nm = 0.5`, `motor_peak_RPM = 3000.0`, gearbox min/max clamps 20/50, efficiency `0.85`
  - `compute_housing()`: moment arm `0.1` (100mm), minimum wall `4.0`, added length margin `+ 80.0`
  - `_export_fusion_csv()`: `flange_t = 8.0`

Many of those are *design assumptions* rather than direct CAD constants, but if ARIA-OS is going to reason about mechanical viability, those assumptions should be centralized and versioned in `context/` rather than embedded in code.

## 7) What would need to change to call these models programmatically from `aria_os/orchestrator.py` after a part is generated?

Conceptually you need two pieces:
1) **Extract/derive inputs for the model from the generated part** (CadQuery geometry or exported STEP), and
2) **Standardize model outputs into an ARIA-OS “validation report”** that the orchestrator can log and decide pass/fail on.

### A) Where to hook in
Right now ARIA-OS calls `validate(code, ...)` in `aria_os/orchestrator.py` and obtains:
- `ValidationResult.geometry` (the CadQuery `result` object)
- `ValidationResult.bbox` (from `BBOX:` print)
- plus STEP validation via `validate_step_file(...)`

So the integration point is immediately after `result = validate(...)` succeeds, while you still have the solid in memory.

### B) Input mapping needed (per model)

#### `dynamic_drop.simulate_drop_test()`
To make this meaningfully “after part generation”, you must map geometry → parameters:
- `rope_k`: derive from rope choice / rope length / compliance (not from CAD)
- absorber parameters: derive from absorber design part geometry and material (not currently represented in ARIA-OS parts)
- `trigger_g`: derive from flyweight clutch geometry (mass distribution + spring preload)

So: without additional CAD parts + a param extraction layer, this can only run with defaults.

#### `static_tests.simulate_static_pawl()`
This is the best candidate for immediate ARIA-OS integration because it accepts explicit geometry overrides:
- From generated pawl/ratchet/housing/shaft parts you’d want to extract:
  - pawl: thickness, tip width, engagement depth, arm length, root height
  - housing: wall thickness at boss
  - shaft: diameter, bearing span

Practical implementation approach:
- Require the generated script to emit a **small JSON “meta” blob** alongside STEP/STL (e.g. `outputs/cad/meta/<part>.json`) containing the named dimensions.
- Then orchestrator can call `simulate_static_pawl(load_steps=[...], pawl_thickness_mm=..., housing_wall_mm=..., shaft_d_mm=...)`.

#### `aria_cem.compute_aria(inputs: ARIAInputs) -> ARIAGeom`
This is a system-level parametric model. Integrating it means:
- You must decide whether the generated CAD part is:
  - conforming to the CEM outputs (CAD should match what CEM derived), or
  - being checked against CEM-derived requirements (CAD can be arbitrary but must not violate constraints)

Either way, you need a translation layer:
- Map context + part metadata → `ARIAInputs`
- Run `compute_aria()`
- Compare CAD extracted dimensions (e.g. drum diameter, housing wall thickness) to computed recommendations (e.g. `BrakeDrumGeom.wall_thickness_mm`)

### C) “What changes would be needed” in code structure (high level)
Without changing any behavior here, the minimal design changes for integration would be:
- Add a new module (e.g. `aria_os/cem_checks.py`) that exposes something like:
  - `run_cem_checks(part_id: str, geometry, step_path: Path, context: dict) -> dict`
- Extend `aria_os/logger.py` to log:
  - model name
  - pass/fail
  - key metrics and thresholds
- Extend `ValidationResult` or add a separate “post-validate checks” object to include:
  - `static_tests_passed`, `dynamic_drop_passed`, `aria_cem_passed`
  - plus numeric metrics (SFs, peak force, arrest distance, etc.)

### D) Key blockers / gaps
- **No reliable dimension extraction from STEP** in current ARIA-OS beyond BBOX. BBOX is insufficient for static_tests inputs like `PAWL_TIP_WIDTH_MM` or `SHAFT_D_MM`.
- **Hardcoded constants** are split across `context/aria_mechanical.md`, `aria_cem.py`, and `static_tests.py`; integration will produce inconsistent results unless constants are unified.
- **Units:** dynamic_drop uses meters internally; static_tests uses mm input parameters but converts to meters; aria_cem mixes mm and m conversions in-line. An integration layer should normalize and assert units explicitly.
